# 测试规格规范

> 本规范适用于软件系统的测试策略设计与测试文档撰写。  
> 规范同时作为 Agent 可执行的规则集使用——每条规则均标注 **REQUIRED** / **PROHIBITED** / **OPTIONAL**，Agent 可机械地对照执行与自检。

## Part 1：术语表

本规范中所有术语以此定义为准，Agent 和人均使用同一词汇：

| 术语 | 定义 |
|------|------|
| **Test Case（测试用例）** | 验证特定功能或行为的输入、执行步骤和预期结果的集合 |
| **Test Suite（测试套件）** | 针对同一模块或功能的测试用例集合 |
| **Fixture（测试固件）** | 测试执行前需要准备的预置数据和环境状态 |
| **Mock** | 模拟外部依赖的测试替身，用于验证交互行为 |
| **Stub** | 返回固定值的测试替身，用于隔离被测单元 |
| **Unit Test（单元测试）** | 测试最小可测单元（通常是函数或方法），所有外部依赖用 Stub/Mock 替代 |
| **Integration Test（集成测试）** | 测试多个模块之间的接口契约，被测模块间使用真实依赖 |
| **E2E Test（端到端测试）** | 模拟真实用户操作路径的完整流程测试 |
| **Test Boundary（测试边界）** | 被测单元与其依赖之间的接口——测试在边界处隔离 |
| **Equivalence Class（等价类）** | 对被测逻辑产生相同行为的输入集合；同一类中的任意一个输入代表整类 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

## Part 2：触发条件

测试文档的更新判断基于核心原则：**测试策略或测试范围发生变更时写文档，测试实现变更时不写**。

**REQUIRED** 新建或更新测试文档的条件（满足任一即触发）：

- 新系统或新子系统从零开始设计（需要测试计划）
- 新增对外暴露接口的模块（接口契约需要集成测试覆盖）
- 测试策略发生变更（引入新的测试层级、测试工具替换、覆盖率目标调整）
- 单次变更涉及 ≥ 3 个模块的集成测试场景（需更新测试计划中的范围）
- 已有文档记录的准入/准出标准被推翻

**PROHIBITED** 触发测试文档更新的变更：

- 单个测试用例的新增或修改（不改变策略）
- 测试实现的重构（测试逻辑不变）
- Bug 修复对应的测试补充（策略范围不变）

## Part 3：测试策略与方法

### 3.1 测试分层原则

测试分三层：单元测试 → 集成测试 → E2E 测试。

**指导原则**：

- 越靠近底层的测试，执行越快、隔离越彻底、数量越多
- 越靠近顶层的测试，覆盖范围越广、维护成本越高、数量越少
- **REQUIRED** 在测试计划中明确每个层级的测试边界（Part 3.2）

**PROHIBITED** 颠倒金字塔：大量 E2E 测试 + 少量单元测试——导致测试套件缓慢且不稳定。

### 3.2 测试边界识别

测试边界决定"这个场景在哪个层级覆盖"。边界识别是测试计划的核心工作。

#### 单元测试边界

- 被测范围是单一函数或方法
- 所有外部依赖（数据库、网络调用、其他模块）用 Stub/Mock 替代
- 判断标准：移除所有依赖后，被测逻辑是否仍然可以独立验证？→ 是则属于单元测试

**PROHIBITED** 在单元测试中使用真实数据库或真实网络连接。

#### 集成测试边界

- 被测范围是两个或多个模块的接口契约
- 断言针对接口输出，而非任一模块的内部状态
- 判断标准：测试目的是验证模块间的数据流转换？→ 是则属于集成测试
- **PROHIBITED** 在集成测试中用 Mock 替代被测模块之间的直接依赖——那会退化为单元测试

#### E2E 测试边界

- 被测范围是完整的用户操作路径（从输入到最终可观测结果）
- **REQUIRED** 仅覆盖 Happy Path（主路径）
- **PROHIBITED** 在 E2E 层覆盖边界条件和错误路径——这些属于单元/集成测试层

#### 等价类划分

将被测函数的所有输入分组：同一组内任意一个输入对系统行为产生相同的效果。每个等价类只需一个代表性测试用例。

**REQUIRED** 划分步骤：

1. 列出所有输入参数及其取值范围
2. 识别有效等价类（系统正常处理的输入）
3. 识别无效等价类（系统应拒绝或报错的输入）
4. 从每个类中选择一个代表性值编写测试用例

**REQUIRED** 每个等价类（有效类和无效类）至少一个测试用例。

#### 边界值分析

系统行为在等价类边界处最容易出错。**REQUIRED** 对数值型和序列型输入，在以下点各设测试用例：

- 最小有效值、最小有效值 - 1
- 最大有效值、最大有效值 + 1
- 0（若在取值范围内）

### 3.3 测试数据管理

**REQUIRED** 在测试计划中声明测试数据策略（选择其一并说明理由）：

- **Fixture**：预定义静态数据，适用于稳定的业务实体
- **Factory**：动态生成数据，适用于需要参数化的测试

**PROHIBITED** 使用生产环境数据作为测试数据。

### 3.4 覆盖率目标

**REQUIRED** 在测试计划中明确记录本项目的目标覆盖率并说明依据。覆盖率目标因项目性质而异，不存在统一默认值。

**REQUIRED** 记录格式：

```
单元测试覆盖率目标：X%（依据：[核心模块边界清晰 / 安全关键路径 / ...]）
集成测试覆盖率目标：X%（依据：[模块间接口数量 / 外部依赖数量 / ...]）
```

**PROHIBITED** 以覆盖率为唯一质量指标——高覆盖率不保证测试质量，Mock 隔离所有真实行为的测试可以达到 100% 覆盖率但验证不了任何有意义的逻辑。

## Part 4：文档规范

### 4.1 文档分类

测试文档分为两类：

| 文档 | 回答的问题 | 存放位置 |
|------|-----------|---------|
| **测试计划** | "测什么、在哪个层级测、用什么标准判断完成" | `docs/test/plan-*.md` |
| **测试用例** | "具体输入是什么、预期输出是什么" | `docs/test/cases-*.md` |

### 4.2 测试计划（Test Plan）

**定位**：描述测试策略、范围和边界的总体文档。

**REQUIRED 章节（按顺序）**：

#### § 测试目标

- 一句话描述测试目的（验证哪个功能/模块的哪个属性）
- **PROHIBITED** 泛化目标（如"确保系统稳定"）

#### § 测试策略

- 各测试层级（Unit/Integration/E2E）的覆盖范围和边界（按 Part 3.2 定义）
- 覆盖率目标（按 Part 3.4 格式记录）
- 测试工具和框架
- **REQUIRED** 对每个层级说明"覆盖哪些场景，原因是什么"——不是"我们会做单元测试"，而是"X 模块的 Y 逻辑在单元测试层覆盖，因为所有依赖可独立 Mock"

#### § 测试范围

- **包含范围**：明确列出要测试的功能模块及测试层级
- **排除范围**：明确列出不测试的内容及原因
- **PROHIBITED** 模糊的边界描述（"主要功能"、"核心路径"）

#### § 测试数据策略

按 Part 3.3 声明测试数据来源（Fixture 或 Factory）及选择理由。

#### § 准入/准出标准

- **准入标准**：开始测试前必须满足的条件
- **准出标准**：测试完成、可进入下一阶段的条件
- **REQUIRED** 所有标准必须可量化验证

---

### 4.3 测试用例（Test Cases）

**定位**：具体可执行的测试场景描述。

**REQUIRED 格式**，每个测试用例包含以下字段：

| 字段 | 说明 | 必填 |
|------|------|------|
| **ID** | 唯一标识符，格式 `TC-{模块}-{序号}` | REQUIRED |
| **标题** | "[动作] + [预期结果]" 格式 | REQUIRED |
| **层级** | Unit / Integration / E2E | REQUIRED |
| **等价类** | 所属等价类（有效类/无效类 + 边界值标注）| REQUIRED |
| **前置条件** | 执行前必须满足的状态 | REQUIRED |
| **输入数据** | 具体的输入值或参数 | REQUIRED |
| **执行步骤** | 详细操作步骤（≤ 10 步，每步单一操作）| REQUIRED |
| **预期结果** | 可机械验证的预期输出（具体值或状态）| REQUIRED |
| **优先级** | P0/P1/P2/P3 | REQUIRED |

**优先级定义**：

- **P0**：失败则阻止发布（核心路径）
- **P1**：失败则严重影响用户体验
- **P2**：常规功能路径
- **P3**：边界条件和错误路径

### 4.4 目录结构与命名

**REQUIRED** 目录结构：

```
docs/test/
├── 00-overview.md             # 测试文档总览（唯一，导航入口）
├── 10-plan-{system}.md        # 测试计划
├── 20-cases-{module}.md       # 测试用例（按模块拆分）
├── deprecated/                # 有替代文档的废弃文档
└── reference/                 # 参考文档
```

**命名规则**：

- 文件名全小写，单词间用连字符 `-` 分隔
- **PROHIBITED** 使用 `v1/`、`v2/`、`old-` 等版本前缀
- **REQUIRED** 同一主题只允许一个 `active` 文档存在
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹；无替代文档则直接删除

### 4.5 文档间关联

**REQUIRED** `00-overview.md` 作为测试文档的导航入口，列出所有 `active` 文档的链接与一句话说明。

**交叉引用规则**：

- **REQUIRED** 测试用例引用关联的功能需求文档
- **PROHIBITED** 复制大段内容，使用相对链接指向原文

## Part 5：写作规则

### 5.1 文档状态标注

**REQUIRED** 每个文档头部包含状态标注：

```markdown
> **Status**: `draft` | `active`
```

废弃文档移入 `deprecated/` 文件夹，文件头部添加：

```markdown
> **Superseded by**: [链接]
```

**状态规则**：

- `draft → active`：自检清单（Part 6）和 `docs-review-spec` 全部通过后标记为 `active`
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹并添加 `Superseded by` 链接；无替代文档则直接删除

### 5.2 语言精确性

**PROHIBITED** 在测试用例中使用模糊词汇：大概、可能、应该、通常、视情况而定。

**REQUIRED** 使用确定性语言：

| ✗ | ✓ |
|---|---|
| "页面应该显示正确" | "页面显示文本'操作成功'" |
| "数据可能被保存" | "数据库中新增一条记录，status = 'active'" |
| "响应时间通常很快" | "响应时间 ≤ 200ms" |

### 5.3 测试用例格式

**标题格式**：

- **REQUIRED** 使用 "[动作] + [预期结果]" 格式
- 示例：`验证用户登录成功后跳转到首页`
- **PROHIBITED**：`测试登录功能`

**执行步骤格式**：

- **REQUIRED** 每条步骤以动词开头，步骤编号连续
- **PROHIBITED** 一条步骤包含多个操作

```markdown
✓
1. 打开登录页面
2. 输入有效用户名
3. 输入有效密码
4. 点击"登录"按钮

✗
1. 打开登录页面并输入用户名密码然后点击登录
```

**预期结果格式**：

- **REQUIRED** 每个预期结果可机械验证，包含具体数据值或状态
- **PROHIBITED** 主观描述（"界面美观"、"操作流畅"）

### 5.4 内容边界

| 内容类型 | 测试文档 | 实现文档 |
|---------|---------|---------|
| 测试策略和范围 | REQUIRED | 可选 |
| 测试边界定义 | REQUIRED | PROHIBITED |
| 测试用例描述 | REQUIRED | PROHIBITED |
| 测试数据策略 | REQUIRED | 可选 |
| 准入/准出标准 | REQUIRED | PROHIBITED |
| 测试代码实现 | PROHIBITED | REQUIRED |
| Mock/Stub 实现细节 | PROHIBITED | REQUIRED |
| 测试执行结果记录 | PROHIBITED | 运营文档 |

## Part 6：自检清单

Agent 在生成或审查测试文档后，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。存在未通过项时，**REQUIRED** 修正对应内容后重新检查。

### 6.1 测试计划

- [ ] 测试目标是具体功能/属性，无泛化描述（无"确保系统稳定"）？
- [ ] 各测试层级的边界用 Part 3.2 规则明确定义？
- [ ] 覆盖率目标已记录并附有依据（不是默认数字）？
- [ ] 准入/准出标准可量化验证？
- [ ] 排除范围每条都有排除原因？
- [ ] 测试数据策略已声明（Fixture 或 Factory）及选择理由？

### 6.2 测试用例

- [ ] 每个用例有唯一 ID（格式 `TC-{模块}-{序号}`）？
- [ ] 标题使用"[动作] + [预期结果]"格式？
- [ ] 层级（Unit/Integration/E2E）已标注且与 Part 3.2 定义一致？
- [ ] 等价类已标注（有效类/无效类 + 边界值）？
- [ ] 执行步骤 ≤ 10 步且每步单一操作？
- [ ] 预期结果可机械验证，包含具体数据？
- [ ] 无模糊词汇（大概、可能、应该等）？
- [ ] 数值型输入覆盖了最小/最大有效值的边界点（Part 3.2 边界值分析）？

### 6.3 通用质量

- [ ] Status 标注存在（`draft` 或 `active`）？
- [ ] 文档间引用使用相对链接，无大段内容复制？
- [ ] `00-overview.md` 导航完整（仅列 `active`）？
- [ ] 无实现细节（测试代码、Mock 实现、工具配置）？

## Agent 执行工作流

Agent 处理测试文档任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不需要写/更新文档，结束
   └── 满足触发条件 → 继续

2. 判断文档类型（Part 4.1）
   ├── 测试策略/范围/边界 → 测试计划（Part 4.2）
   └── 具体测试场景 → 测试用例（Part 4.3）

3. 识别测试边界（Part 3.2）
   ├── 确定各场景所属层级（Unit/Integration/E2E）
   ├── 进行等价类划分（有效类 + 无效类）
   └── 确定边界值测试点

4. 按对应章节结构写作
   ├── 遵守语言精确性规则（Part 5.2）
   ├── 使用正确的测试用例格式（Part 5.3）
   └── 遵守内容边界规则（Part 5.4）

5. 执行自检清单（Part 6）
   ├── 所有项通过 → 继续
   └── 存在未通过项 → 修正后重新检查

6. 执行交付前质量门（docs-review-spec）
   ├── 读取 references/docs-review-spec.md
   ├── 执行合规检查与润色优化
   ├── 所有项通过 → 输出文档
   └── 存在未通过项 → 修正后重新检查
```
