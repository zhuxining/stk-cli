# 产品蓝图规范

> 本规范适用于产品定位、核心命题和长期演进方向的定义。  
> 规范同时作为 Agent 可执行的规则集使用——每条规则均标注 **REQUIRED** / **PROHIBITED** / **OPTIONAL**，Agent 可机械地对照执行与自检。

## Part 1：术语表

本规范中所有术语以此定义为准，Agent 和人均使用同一词汇：

| 术语 | 定义 |
|------|------|
| **Product Blueprint（产品蓝图）** | 描述产品为什么存在、为谁创造价值、核心命题是什么、长期如何演进的顶层文档 |
| **Product Proposition（产品命题）** | 产品对目标用户作出的核心价值判断，用一句确定性陈述表达 |
| **Target User（目标用户）** | 产品优先服务的人群或角色，必须有明确场景和问题 |
| **Core Scenario（核心场景）** | 目标用户反复遇到、产品必须服务的关键使用情境 |
| **Value Boundary（价值边界）** | 产品明确承诺解决和明确不解决的问题边界 |
| **Product Principle（产品原则）** | 指导 PRD、交互和架构取舍的产品层规则 |
| **Evolution Direction（演进方向）** | 产品长期能力扩展的方向，不等同于版本计划或功能清单 |
| **Non-goal（非目标）** | 产品明确不追求的方向，用于防止范围漂移 |
| **Candidate Direction（候选方向）** | 蓝图讨论中形成的可选产品方向，尚未写入正式蓝图 |
| **Load-bearing Assumption（关键假设）** | 若被证明不成立，则核心命题、目标用户或长期方向需要改变的前提 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

## Part 2：触发条件

产品蓝图的更新判断基于一条核心原则：**产品定位、核心命题或长期方向发生变更时写文档，功能细节和实现细节变更时不写**。

**REQUIRED** 新建或更新产品蓝图的条件（满足任一即触发）：

- 新产品或新产品线从零开始定义
- 产品核心命题发生变更
- 目标用户或核心场景发生变更
- 长期演进方向发生变更
- 多个 Feature 需要统一到同一产品方向下
- PRD 或架构文档缺少上层产品依据，导致取舍无法判断
- 已有蓝图中记录的产品原则或非目标被推翻

**PROHIBITED** 触发产品蓝图更新的变更：

- 单个 Feature 的用户交互、验收标准或范围调整
- 纯技术方案、接口契约、数据库结构或模块边界调整
- 测试策略、测试用例或准入/准出标准调整
- 视觉样式、品牌文案或营销页面调整（未改变产品命题）
- Bug 修复或性能优化（未改变产品定位）

## Part 3：蓝图分析方法

### 3.0 蓝图讨论工作流

蓝图讨论用于将粗糙产品想法收敛为可写入产品蓝图的判断。满足 Part 2 触发条件且尚未有明确蓝图结论时，Agent 在正式写作前 **REQUIRED** 执行以下流程。

#### 现状读取

**REQUIRED** 在提出判断前读取已有上下文：

- 若仓库已有 `docs/blueprint/`，读取直接相关的 `active` 蓝图
- 若讨论涉及已有 Feature，读取对应 PRD
- 若讨论涉及技术可行性或系统边界，读取对应架构文档
- 若无相关文档，记录"未发现相关现有文档"，继续讨论

**PROHIBITED** 未读取已有蓝图、PRD 或架构文档就推翻既有产品命题。

#### 问题重述

**REQUIRED** 将用户的初始想法重述为一个产品问题，而不是功能请求：

```markdown
我们要判断的问题是：[目标用户] 在 [核心场景] 中的 [核心问题] 是否值得由产品长期解决。
```

**PROHIBITED** 直接把用户的初始想法改写成 Feature 名称或实现方案。

#### 关键澄清

**REQUIRED** 在信息不足时提出 3 ~ 5 个澄清问题。问题必须优先覆盖：

- 谁是 Primary User
- 用户成功状态是什么
- 为什么现在需要解决
- 已有替代方案是什么
- 关键约束是什么（时间、资源、合规、技术边界）

**PROHIBITED** 在 Primary User 和成功状态不清楚时写入正式蓝图。

#### 发散方向

**REQUIRED** 生成 3 ~ 5 个 Candidate Direction。每个方向包含：

- 方向名称
- 服务的目标用户
- 解决的核心问题
- 为什么不同于用户的初始方案或已有方向
- 明确不做什么

可使用以下发散视角，但 **PROHIBITED** 机械套用全部视角：

| 视角 | 问题 |
|------|------|
| 反向思考 | 如果做相反选择，会暴露什么真实约束？ |
| 简化版本 | 10 倍更简单的产品形态是什么？ |
| 约束移除 | 若暂时移除时间、预算或技术约束，最强命题是什么？ |
| 用户切换 | 若换成另一类用户，产品命题是否更清晰？ |
| 规模放大 | 若使用量扩大 10 倍，哪个价值或问题会变得更明显？ |

#### 收敛评估

**REQUIRED** 对每个 Candidate Direction 使用以下维度评估：

| 维度 | 判断问题 |
|------|----------|
| 用户价值 | 这是强痛点还是弱改进？用户不解决会付出什么代价？ |
| 命题契合 | 该方向是否强化产品核心命题，还是稀释方向？ |
| 可持续性 | 该方向是否能形成长期演进空间，而非一次性 Feature？ |
| 差异化 | 用户为什么选择本产品，而不是继续使用现有替代方案？ |
| 代价 | 该方向会引入什么产品、组织或技术负担？ |

**REQUIRED** 给出一个推荐方向。只有当第二选择与推荐方向存在接近的真实取舍时，才列出一个替代方向。

**PROHIBITED** 输出平均主义选项列表，或把所有方向都标为可行。

#### 假设与攻击面

**REQUIRED** 明确推荐方向的关键假设：

- **关键假设**：当前判断依赖什么前提
- **验证方式**：如何用访谈、原型、数据或已有使用行为验证
- **失效后果**：假设不成立时，蓝图中哪部分需要改变

当讨论涉及外部依赖、高并发、数据迁移或高切换成本时，**REQUIRED** 额外检查以下攻击面：

| 攻击面 | 问题 |
|--------|------|
| 依赖失败 | 外部依赖不可用时，产品命题是否仍成立？ |
| 规模放大 | 使用量或数据量扩大 10 倍时，哪个承诺最先失效？ |
| 回滚成本 | 方向错误后，产品、数据或用户心智能否回到旧状态？ |

若攻击面击穿推荐方向，**REQUIRED** 修改推荐方向或明确放弃该方向。

#### 推荐结论

正式写入蓝图前，**REQUIRED** 输出蓝图候选结论：

```markdown
## 蓝图候选结论

### 推荐方向
[一个明确推荐方向及理由]

### 不推荐方向
[明确不采用的方向及理由]

### 关键假设
- [假设]；验证方式：[方式]；失效后果：[后果]

### 非目标
- [不做什么]；理由：[理由]

### 写入蓝图的内容
- 产品定位：[候选表达]
- 核心命题：[候选表达]
- 长期演进方向：[候选方向]
```

**REQUIRED** 用户确认蓝图候选结论后，才将内容写入正式产品蓝图。用户未确认时继续讨论，不更新蓝图文档。

### 3.1 产品定位

产品定位回答："这个产品为什么必须存在？"

**REQUIRED** 产品定位包含以下四项：

- **目标用户**：优先服务的人群或角色
- **核心问题**：用户在核心场景中反复遇到的问题
- **核心方式**：产品用什么能力类别解决问题（不写实现方案）
- **长期价值**：持续使用后形成的用户或业务价值

**REQUIRED** 使用一句话定位格式：

```markdown
[产品名] 为 [目标用户] 在 [核心场景] 中解决 [核心问题]，通过 [核心方式] 形成 [长期价值]。
```

**PROHIBITED** 内容：

- 功能清单（"支持 A、B、C"）
- 营销口号（无法判断取舍的宣传语）
- 技术实现（模型、框架、数据库、接口）
- 竞品比较细节（价格、功能表、市场份额）

### 3.2 目标用户与场景

**REQUIRED** 将用户分为三类：

| 类型 | 定义 | 写作要求 |
|------|------|----------|
| **Primary User** | 产品优先服务、决策时优先满足的人群 | 1 ~ 3 类，每类包含角色、核心问题、成功状态 |
| **Secondary User** | 会受益但不驱动核心取舍的人群 | 可选，必须说明低于 Primary User 的原因 |
| **Excluded User** | 明确不服务的人群 | REQUIRED，必须说明排除原因 |

**Core Scenario** 的合格标准：

- 描述用户在什么情境下遇到什么问题
- 能解释为什么该问题值得产品长期解决
- 不等同于一个具体功能或页面

**PROHIBITED** 用泛化人群代替目标用户，如"所有用户"、"普通用户"、"企业客户"。

### 3.3 核心命题

核心命题是产品蓝图的中心判断。**REQUIRED** 每份产品蓝图只有一个核心命题。

合格的核心命题满足：

1. **可指导取舍**：能判断某个 Feature 是否偏离产品方向
2. **非功能清单**：不是把多个功能串成一句话
3. **有边界**：隐含或显式说明产品不追求什么
4. **稳定**：不会因单个版本或单个 Feature 变化而失效

**REQUIRED** 使用以下格式：

```markdown
本产品的核心命题是：[一句确定性陈述]。
```

**PROHIBITED** 核心命题包含验收标准、交互步骤、接口名称、模型名称、数据库或测试策略。

### 3.4 产品原则

产品原则用于指导后续 PRD、交互和架构取舍。**REQUIRED** 每份蓝图包含 3 ~ 5 条产品原则。

每条原则 **REQUIRED** 包含：

- **名称**：短语
- **规则**：做决策时遵守的确定性规则
- **理由**：为什么该规则服务于核心命题

**REQUIRED** 格式：

```markdown
- **[原则名称]**：[规则]。理由：[理由]。
```

**PROHIBITED** 内容：

- 无法指导取舍的价值词（"简单"、"高效"、"智能"）
- 超过 5 条原则
- 与实现方案绑定的原则

### 3.5 长期演进方向

长期演进方向回答："产品沿哪些方向变强，而不是下一版本做哪些功能？"

**REQUIRED** 记录 2 ~ 4 个演进方向，每个方向包含：

- **方向名称**
- **能力变化**：产品长期会增强哪类能力
- **价值理由**：为什么该方向服务于核心命题
- **边界**：该方向不扩展到什么范围

**PROHIBITED** 内容：

- 版本计划、排期、里程碑日期
- Feature backlog
- 逐项验收标准
- 技术路线图或底层实现

### 3.6 非目标

Non-goal 用于阻止产品范围漂移。**REQUIRED** 每份蓝图包含 3 ~ 7 条非目标。

每条 Non-goal **REQUIRED** 包含：

- 不追求什么
- 为什么不追求
- 若相关需求出现，应归入哪个文档或决策流程

**PROHIBITED** 使用"暂不支持"、"以后考虑"等无决策信息的表述。

## Part 4：文档规范

### 4.1 文档分类

产品蓝图文档分为两类：

| 文档 | 回答的问题 | 存放位置 |
|------|-----------|---------|
| **总览蓝图** | 整个产品为什么存在、核心命题是什么、长期方向是什么 | `docs/blueprint/00-overview.md` |
| **产品域蓝图** | 某个产品域如何继承总览蓝图并形成自己的边界 | `docs/blueprint/NN-{product-area}.md` |

**判断依据**：

- 影响整个产品的定位、核心命题或长期方向 → 总览蓝图
- 只影响某个产品域的目标用户、场景或原则 → 产品域蓝图
- 只影响单个 Feature → PRD，不写产品蓝图

### 4.2 总览蓝图（Overview Blueprint）

**定位**：产品蓝图体系的入口，描述产品级的存在理由、核心命题和长期方向。

**REQUIRED 章节（按顺序）**：

#### § 产品定位

- 使用 Part 3.1 的一句话定位格式
- 补充 1 ~ 3 句话解释产品存在理由
- **PROHIBITED**：功能清单、技术路线、市场宣传语

#### § 目标用户与场景

- 按 Part 3.2 列出 Primary User、Secondary User（可选）、Excluded User
- 列出 2 ~ 5 个 Core Scenario
- **PROHIBITED**：用户故事、交互流程、页面描述

#### § 核心命题

- 使用 Part 3.3 的固定格式
- 说明该命题如何排除不相关方向
- **PROHIBITED**：多个并列核心命题

#### § 产品原则

- 按 Part 3.4 记录 3 ~ 5 条原则
- **REQUIRED** 每条原则能指导 PRD 或架构取舍

#### § 长期演进方向

- 按 Part 3.5 记录 2 ~ 4 个方向
- **PROHIBITED**：版本计划、日期、Feature backlog

#### § 非目标

- 按 Part 3.6 记录 3 ~ 7 条 Non-goal
- **REQUIRED** 每条包含排除理由

#### § 文档关联

- 链接主要 PRD、架构文档和测试计划
- **PROHIBITED** 复制其他文档内容

### 4.3 产品域蓝图（Product Area Blueprint）

**定位**：描述某个产品域如何继承总览蓝图，并声明该产品域的价值边界。

**REQUIRED 章节（按顺序）**：

#### § 继承关系

- 引用总览蓝图
- 说明本产品域继承哪些核心命题和产品原则

#### § 产品域定位

- 使用一句话说明该产品域服务的目标用户、核心场景和价值边界
- **PROHIBITED**：功能清单、交互细节、接口契约

#### § 场景边界

- 列出本产品域覆盖的 Core Scenario
- 列出本产品域明确不覆盖的场景及理由

#### § 产品域原则

- 记录本产品域特有原则，≤ 3 条
- 若无特有原则，省略此节

#### § 关联文档

- 链接该产品域下的 PRD、架构文档和测试计划
- **PROHIBITED** 复制其他文档内容

### 4.4 目录结构与命名

**REQUIRED** 目录结构：

```
docs/blueprint/
├── 00-overview.md              # 总览蓝图（唯一，导航入口）
├── NN-product-area.md          # 产品域蓝图（两位数字序号，表示阅读顺序）
├── deprecated/                 # 有替代文档的废弃文档
└── reference/                  # 参考材料
```

**命名规则**：

- 文件名全小写，单词间用连字符 `-` 分隔
- 数字编号仅表示阅读顺序，不表示版本
- **PROHIBITED** 使用 `v1/`、`v2/`、`old-` 等版本标记
- **REQUIRED** 同一主题只允许一个 `active` 文档存在
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹；无替代文档则直接删除

### 4.5 文档间关联

**REQUIRED** `00-overview.md` 作为产品蓝图的导航入口，列出所有 `active` 产品域蓝图的链接与一句话说明。`deprecated/` 文件夹中的文档不出现在导航中。

**交叉引用规则**：

- **REQUIRED** 产品域蓝图引用总览蓝图
- **REQUIRED** PRD 引用对应产品蓝图，说明 Feature 继承的产品命题或原则
- **OPTIONAL** 架构文档引用产品蓝图，说明关键技术取舍对应的产品原则
- **PROHIBITED** 在产品蓝图中复制 PRD、架构或测试文档内容

## Part 5：写作规则

### 5.1 文档状态标注

**REQUIRED** 每个文档头部包含状态标注：

```markdown
> **Status**: `draft` | `active`
```

废弃文档移入 `deprecated/` 文件夹，文件头部添加：

```markdown
> **Superseded by**: [链接]
```

**状态规则**：

- `draft → active`：自检清单（Part 6）和 `docs-review-spec` 全部通过后标记为 `active`
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹并添加 `Superseded by` 链接；无替代文档则直接删除

### 5.2 语言精确性

**PROHIBITED** 在蓝图中使用模糊词汇：可能、大概、应该、通常、视情况而定、尽量、适当、未来再说。

**REQUIRED** 使用确定性语言：

| ✗ | ✓ |
|---|---|
| "产品应该服务更多用户" | "产品优先服务独立开发者和小型团队，不服务大型企业采购流程" |
| "后续可能增加协作能力" | "长期演进方向之一是多人协作，边界是不提供项目管理套件" |
| "保持简单易用" | "默认路径中用户完成核心任务的概念数量 ≤ 3 个" |

### 5.3 内容边界

下表定义产品蓝图与其他文档的边界：

| 内容类型 | 产品蓝图 | PRD | 架构文档 | 测试文档 |
|---------|----------|-----|----------|----------|
| 产品存在理由 | REQUIRED | 可引用 | PROHIBITED | PROHIBITED |
| 核心命题 | REQUIRED | 可引用 | 可引用 | PROHIBITED |
| 长期演进方向 | REQUIRED | 可引用 | 可引用 | PROHIBITED |
| 产品原则 | REQUIRED | 可引用 | 可引用 | PROHIBITED |
| 目标用户与核心场景 | REQUIRED | REQUIRED（具体到 Feature） | PROHIBITED | 可引用 |
| 用户可见功能 | PROHIBITED | REQUIRED | 可选 | 可引用 |
| 交互流程 | PROHIBITED | REQUIRED | PROHIBITED | 可引用 |
| 验收标准 | PROHIBITED | REQUIRED | PROHIBITED | 可引用 |

## Part 6：自检清单

Agent 在生成或审查产品蓝图后，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。存在未通过项时，**REQUIRED** 修正对应内容后重新检查。

### 6.1 内容完整性

- [ ] 正式写作前已执行 Part 3.0 蓝图讨论工作流，或已有明确蓝图结论？
- [ ] 产品定位使用 Part 3.1 的一句话格式？
- [ ] 目标用户分为 Primary / Secondary（可选）/ Excluded，且 Excluded 有排除理由？
- [ ] 只有一个核心命题？
- [ ] 产品原则为 3 ~ 5 条，且每条包含名称、规则、理由？
- [ ] 长期演进方向为 2 ~ 4 个，且无版本计划或 Feature backlog？
- [ ] 非目标为 3 ~ 7 条，且每条包含排除理由？
- [ ] 文档关联指向 PRD、架构或测试文档，未复制内容？

### 6.2 质量检查

- [ ] 若本次新增或改变蓝图结论，蓝图候选结论已获得用户确认？
- [ ] 关键假设包含验证方式和失效后果？
- [ ] Status 标注存在（`draft` 或 `active`）？
- [ ] 无模糊词汇（可能、大概、应该、通常等）？
- [ ] 无验收标准、用户交互细节、接口契约、数据库细节或测试用例？
- [ ] 核心命题能指导 Feature 取舍？
- [ ] 产品原则能指导 PRD 或架构取舍？
- [ ] `00-overview.md` 导航与文档状态一致（仅列 `active`）？

## Agent 执行工作流

Agent 处理产品蓝图任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不需要写/更新产品蓝图，转向 PRD/架构/测试规范或结束
   └── 满足触发条件 → 继续

2. 判断文档类型（Part 4.1）
   ├── 产品级定位/核心命题/长期方向 → 总览蓝图（Part 4.2）
   └── 单一产品域的定位/场景/边界 → 产品域蓝图（Part 4.3）

3. 执行蓝图讨论工作流（Part 3.0）
   ├── 读取现有蓝图、PRD、架构文档
   ├── 重述产品问题，提出关键澄清
   ├── 生成并评估 Candidate Direction
   ├── 输出蓝图候选结论、关键假设、非目标
   ├── 用户确认 → 继续
   └── 用户未确认 → 继续讨论，不写入蓝图

4. 完成蓝图分析
   ├── 使用 Part 3.1 定义产品定位
   ├── 使用 Part 3.2 明确目标用户与核心场景
   ├── 使用 Part 3.3 定义唯一核心命题
   ├── 使用 Part 3.4 形成产品原则
   ├── 使用 Part 3.5 记录长期演进方向
   └── 使用 Part 3.6 记录非目标

5. 按对应章节结构写作
   ├── 遵守语言精确性规则（Part 5.2）
   ├── 遵守内容边界规则（Part 5.3）
   └── 建立文档关联（Part 4.5）

6. 执行自检清单（Part 6）
   ├── 所有项通过 → 继续
   └── 存在未通过项 → 修正后重新检查

7. 执行交付前质量门（docs-review-spec）
   ├── 读取 references/docs-review-spec.md
   ├── 执行合规检查与润色优化
   ├── 所有项通过 → 输出文档
   └── 存在未通过项 → 修正后重新检查
```
