# 文档审校与润色规范

> 本规范适用于产品蓝图、架构设计文档、功能需求规格和测试规格的交付前质量检查与润色。  
> 它不是新的项目文档类型，不创建 `docs/review/` 目录；它是所有设计文档首次成稿后的质量门。

## Part 1：术语表

| 术语 | 定义 |
|------|------|
| **Docs Review（文档审校）** | 对文档结构、内容边界、状态标注、链接、术语和禁止内容进行验证 |
| **Polish（润色）** | 在不改变核心意义和关键信息的前提下，优化结构连贯性、语言精确性、文体语气、清晰简洁性、标点语法 |
| **Target Text（目标文本）** | 用户明确要求检查或润色的文本，或 Agent 首次生成的设计文档 |
| **Document Contract（文档契约）** | 目标文档所属规范定义的章节结构、内容边界、写作规则和自检清单 |
| **Full Document Review（完整文档审校）** | 对一个完整设计文档执行合规检查与润色，必须读取对应文档规范 |
| **Fragment Polish（片段文本润色）** | 对用户粘贴的段落、章节或短文本执行保意润色，不要求完整文档结构 |
| **Blocking Issue（阻塞问题）** | 会导致文档不应交付或不能从 `draft` 标记为 `active` 的问题 |
| **Non-blocking Issue（非阻塞问题）** | 不影响文档契约成立，但会降低表达质量、可读性或一致性的问题 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

---

## Part 2：触发条件

**REQUIRED** 执行本文档规范的条件（满足任一即触发）：

- Agent 首次写完任一设计文档后
- 用户显式要求"润色"、"检查"、"优化"、"改稿"、"去 AI 味"、"审一下"
- 文档准备从 `draft` 转为 `active` 前
- 用户要求检查设计文档是否符合 `code-docs-spec`

**PROHIBITED** 用本文档规范替代原文档类型规范。执行 Full Document Review 前必须先判断目标文档属于：

- 产品蓝图 → `blueprint-spec.md`
- 架构设计文档 → `architecture-spec.md`
- 功能需求规格 → `prd-spec.md`
- 测试规格 → `test-spec.md`

## Part 3：Pre-flight

### 3.1 确认目标文本

**REQUIRED** 明确 Target Text 来源：

- 若用户指定文件路径，读取该文件
- 若用户粘贴文本，以粘贴内容为目标文本
- 若是 Agent 首次成稿，以刚生成的文档为目标文本

**PROHIBITED** 在没有明确 Target Text 时猜测要润色的内容。若用户只说"润色一下"但没有文本或文件路径，必须要求用户提供目标文本。

### 3.2 判断处理模式

**REQUIRED** 先判断任务属于哪种模式：

| 模式 | 触发条件 | 执行要求 |
|------|----------|----------|
| **Full Document Review** | 目标是仓库文件、完整 Markdown 文档，或包含 Status 与完整章节结构 | 读取对应文档规范，执行合规检查与润色优化 |
| **Fragment Polish** | 目标是用户粘贴的段落、章节、说明文字，或不具备完整文档结构的文本 | 不要求 Status、导航入口或完整章节结构；只检查原意保留、语气、清晰度、术语一致性和标点语法 |

若无法判断模式，**REQUIRED** 先询问用户目标是"完整文档审校"还是"片段文本润色"。

### 3.3 判断文档类型

Full Document Review **REQUIRED** 根据路径、标题、章节结构或用户说明判断文档类型：

| 目标文档 | 对应规范 |
|----------|----------|
| `docs/blueprint/*.md` | `references/blueprint-spec.md` |
| `docs/architecture/*.md` | `references/architecture-spec.md` |
| `docs/prd/*.md` | `references/prd-spec.md` |
| `docs/test/*.md` | `references/test-spec.md` |

Full Document Review **REQUIRED** 读取对应规范后再检查或润色，避免只改语言而破坏文档契约。

Fragment Polish **OPTIONAL** 在文本明确属于某类设计文档时参考对应规范的语言和术语规则，但 **PROHIBITED** 因片段缺少完整章节、Status 或导航入口而判为不通过。

### 3.4 锁定读者与语气

**REQUIRED** 从目标文本推断目标读者和语气：

- 产品蓝图：产品负责人、核心设计者、跨职能决策者；语气应明确、克制、可指导取舍
- PRD：产品、设计、研发、测试；语气应具体、可验证、无歧义
- 架构文档：研发、架构评审者、维护者；语气应精确、边界清晰、避免市场化表达
- 测试文档：测试、研发、发布负责人；语气应可执行、可验证、步骤明确

若读者和语气无法从文本或路径推断，**REQUIRED** 先询问用户，不能自行选择风格。

## Part 4：两段式流程

### 4.1 合规检查

合规检查先于润色。Full Document Review **REQUIRED** 按以下顺序执行：

1. **章节结构**：目标文档是否包含所属规范要求的 REQUIRED 章节
2. **内容边界**：是否混入其他文档类型的内容
3. **状态标注**：是否包含 `> **Status**:`，且取值为 `draft` 或 `active`
4. **链接与引用**：相对链接是否存在，是否复制其他文档的大段内容
5. **术语一致性**：同一概念是否使用同一名称
6. **禁止内容**：是否包含所属规范明确 PROHIBITED 的内容

**REQUIRED** 将问题分为：

- **Blocking Issue**：违反文档契约、缺少 REQUIRED 章节、混入禁止内容、状态标注错误、关键链接缺失
- **Non-blocking Issue**：表达重复、段落顺序不顺、语气不统一、标点或格式问题

存在 Blocking Issue 时，**REQUIRED** 先修正或向用户报告缺失信息，**PROHIBITED** 只做语言润色后交付。

Fragment Polish 不执行完整文档合规检查；只检查以下项目：

- 是否保留原文核心意义与关键信息
- 是否引入新的业务事实、承诺、验收标准或实现细节
- 术语、语气、标点和段落是否一致
- 句子是否更清晰、更简洁且无歧义

### 4.2 润色优化

通过合规检查或完成阻塞问题修正后，按以下五类执行润色：

#### 结构与连贯性改进

- **REQUIRED** 保证段落内部及段落间逻辑衔接自然流畅
- **REQUIRED** 优化句子顺序，提升条理性和逻辑清晰度
- **PROHIBITED** 静默重排章节或合并大段内容；若大段重组必要，必须说明原因

#### 语言精准与多样性提升

- **REQUIRED** 用更精确的词替换含糊、重复或无法验证的表达
- **REQUIRED** 在不牺牲清晰度的前提下调整句式，改善阅读节奏
- **PROHIBITED** 为了"高级"或"生动"引入新含义、新承诺或新事实

#### 文体与语气调整

- **REQUIRED** 保持全文语气统一，符合目标读者预期
- **REQUIRED** 根据文档类型把握正式、亲切、专业或可执行的尺度
- **PROHIBITED** 将设计文档改写成营销文案、教程、会议纪要或实现说明

#### 清晰与简洁性加强

- **REQUIRED** 删除冗余表达，但不得删除需求、约束、链接、数字、状态、范围边界或 Non-goal
- **REQUIRED** 修正易产生歧义的句子
- **PROHIBITED** 静默删除看似重复但承载不同判断的信息

#### 标点与语法规范

- **REQUIRED** 检查并修正拼写、标点、用词和语法错误
- **REQUIRED** 优化段落划分，提升视觉层次
- **REQUIRED** 中文文档使用中文标点；中英文术语混排时保持空格和术语一致

## Part 5：硬规则

**REQUIRED** 遵守以下规则：

- 核心意义与关键信息完整无损
- 数字、阈值、状态、链接、In Scope、Out of Scope、Non-goal、验收标准、接口名称、实体名称不得静默删除或改写
- 润色前后仍符合原文档类型的内容边界
- 若用户给出修改清单，先对照目标文本确认每项是否存在
- 文件修改后，重新阅读每处编辑附近内容，检查孤立段落、残留字符、术语不一致和链接断裂

**PROHIBITED** 以下行为：

- 凭空补业务事实、验收标准、接口契约、数据库细节或测试用例
- 为了语言流畅改变产品命题、需求范围、架构决策或测试边界
- 在用户未要求删除时静默删除完整句子、段落或章节
- 不读取对应规范就直接改稿
- 输出无关解释、客套话或"希望这有帮助"类结尾

## Part 6：输出规则

### 6.1 文件修改场景

若目标是仓库文件：

- **REQUIRED** 交付润色后的文档
- **REQUIRED** 简要说明 Blocking Issue 是否已修正
- **REQUIRED** 说明仍需用户补充的信息或剩余风险（如有）
- **PROHIBITED** 输出长篇逐句解释，除非用户明确要求

### 6.2 粘贴文本场景

若用户只提供一段文本：

- **REQUIRED** 返回润色后的文本
- **OPTIONAL** 在文本后用一句话说明存在多个可选版本或原文信息不足
- **PROHIBITED** 在润色文本前后添加无关说明

### 6.3 内容缺口场景

若发现必须补充的信息缺失：

- **REQUIRED** 先列出缺失信息和原因
- **PROHIBITED** 编造缺失内容
- **OPTIONAL** 给出可直接填写的占位模板，但不得将占位模板当成最终文档

## Part 7：自检清单

Agent 在交付检查或润色结果前，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。

### 7.1 合规检查

- [ ] Target Text 明确存在？
- [ ] 已判断处理模式（Full Document Review 或 Fragment Polish）？
- [ ] 若模式为 Full Document Review：已判断文档类型、读取对应规范，并检查章节结构、内容边界和 Status？
- [ ] 若模式为 Fragment Polish：未因缺少完整章节、Status 或导航入口而误判不通过？
- [ ] 链接、数字、状态、范围边界、Non-goal、验收标准等关键信息未丢失？

### 7.2 润色质量

- [ ] 段落内部及段落间逻辑衔接自然？
- [ ] 句子顺序清晰，未破坏原意？
- [ ] 语气与目标读者匹配？
- [ ] 冗余表达已删减，歧义表达已修正？
- [ ] 标点、语法、术语、段落划分一致？
- [ ] 未凭空新增业务事实、接口契约、数据库细节或测试用例？

## Agent 执行工作流

Agent 处理文档检查或润色任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不执行 docs-review-spec
   └── 满足触发条件 → 继续

2. 执行 Pre-flight（Part 3）
   ├── 确认 Target Text
   ├── 判断处理模式
   ├── Full Document Review → 判断文档类型并读取对应规范
   ├── Fragment Polish → 不要求完整文档结构
   └── 锁定目标读者与语气

3. 执行合规检查（Part 4.1）
   ├── Full Document Review → 检查文档契约
   ├── Fragment Polish → 检查原意保留与术语一致性
   ├── 存在 Blocking Issue → 修正或向用户报告缺失信息
   └── 无 Blocking Issue → 继续

4. 执行润色优化（Part 4.2）
   ├── 改善结构与连贯性
   ├── 提升语言精准度
   ├── 统一文体与语气
   ├── 加强清晰与简洁性
   └── 修正标点与语法

5. 执行自检清单（Part 7）
   ├── 所有项通过 → 交付
   └── 存在未通过项 → 修正后重新检查
```
