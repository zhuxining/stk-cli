# 架构设计与文档规范

> 本规范适用于任何软件系统的架构设计与文档撰写。  
> 规范同时作为 Agent 可执行的规则集使用——每条规则均标注 **REQUIRED** / **PROHIBITED** / **OPTIONAL**，Agent 可机械地对照执行与自检。

## Part 1：术语表

本规范中所有术语以此定义为准，Agent 和人均使用同一词汇：

| 术语 | 定义 |
|------|------|
| **Module（模块）** | 拥有单一变更理由的代码/逻辑单元 |
| **Boundary（边界）** | 两个模块之间的接口契约；跨越边界必须通过显式接口 |
| **Entity（实体）** | 具有业务身份、生命周期独立的核心对象，不等同于所有数据模型 |
| **Constraint（约束）** | 不可违背的限制，违背导致系统失效或违规 |
| **Principle（原则）** | 指导决策的价值取向，可在权衡中排定优先级 |
| **Decision（决策）** | 在多个可行方案中的有意选择，必须记录理由 |
| **Service（服务）** | 无状态的业务逻辑单元，接受输入、产生输出、不持有可变状态 |
| **ADR** | Architecture Decision Record，单条设计决策的结构化记录 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

## Part 2：触发条件

架构文档的更新判断基于一条核心原则：**接口契约或设计决策发生变更时写文档，实现细节变更时不写**。

**REQUIRED** 新建或更新架构文档的条件（满足任一即触发）：

- 新系统或新子系统从零开始设计
- 新增 Module 且对外暴露接口
- 核心实体的定义或关系发生变更
- 单次变更跨越 ≥ 3 个模块的边界（以对外接口是否被调用或变更为准，非文件数量）
- 安全边界发生变更（信任模型、权限模型、加密方案）
- 已有文档中记录的设计决策被推翻

**PROHIBITED** 触发架构文档更新的变更——未改变接口契约的模块内部变更（实现优化、Bug 修复、配置调整、测试修改）。

## Part 3：设计方法

### 3.1 边界识别

边界划分是架构设计的核心工作。判断两个功能是否应在同一 Module 内，依据以下维度：

**正确的划分依据：**

1. **变更理由一致性**：同一原因驱动变更的代码放在同一 Module。若 A 和 B 因不同原因而需要分别改动，则它们属于不同 Module。
2. **数据所有权**：每份数据只有一个 Module 拥有写权限。哪个 Module 创建并维护这份数据，边界就在哪里。
   - **读权限**：其他模块读取数据 **REQUIRED** 通过所有者模块的接口，**PROHIBITED** 直接访问所有者的存储
   - **派生数据**：模块 B 从模块 A 的数据计算出新数据，派生数据归模块 B 所有，但模块 B **REQUIRED** 通过 A 的接口获取源数据
   - **事件驱动的数据复制**：模块 B 消费模块 A 的事件并写入自己的存储，该副本归模块 B 所有，模块 B 负责副本的一致性
3. **部署独立性**：可以独立部署的单元天然构成边界。
4. **信任级别**：不同信任域（如用户输入 vs 系统内部、外部 API vs 内部服务）之间必须有显式边界。

**PROHIBITED 划分方式：**

- 按功能"看起来相似"划分——相似不代表同一变更理由
- 按技术分层划分（如 controller / service / repository 各为一个 Module）——这是实现结构，不是业务边界
- 按代码文件数量或行数均分

**边界的表达方式：**

边界通过显式接口（函数签名、trait、API 契约）表达，**PROHIBITED** 通过共享全局状态或直接访问另一 Module 的内部数据跨越边界。

#### 依赖方向约束

**REQUIRED** 模块间依赖构成有向无环图（DAG），**PROHIBITED** 循环依赖。

依赖方向规则：

1. **高层不依赖低层实现**：业务策略模块定义接口（trait），基础设施模块实现该接口。依赖指向接口定义方，非实现方
2. **接口所有权**：边界接口由**调用方**（或其所在层的公共契约）定义，被调用方实现。当两个模块需要解耦时，接口定义归高层模块所有
3. **单向数据流**：数据从外层（输入）流向内层（业务核心），再从内层流向外层（输出）。内层模块 **PROHIBITED** 主动调用外层模块

判断依赖方向是否正确的方法：如果移除模块 A 需要修改模块 B 的代码，则 B 依赖 A。检查所有此类关系是否形成 DAG。

#### 跨切关注点

认证、日志、指标、缓存、错误翻译等关注点天然跨越模块边界，不适合归入单一 Module。

处理规则：

- 跨切关注点不构成独立的业务 Module，**PROHIBITED** 将其建模为与业务模块平级的 Module
- **REQUIRED** 在总览文档的"边界划分"章节中声明所有跨切关注点及其实现策略（中间件、trait 默认实现、装饰器等）
- 跨切关注点的实现代码放在独立的基础设施包中，业务模块通过依赖注入或组合方式使用，**PROHIBITED** 在业务模块中硬编码跨切逻辑

#### 接口演进策略

模块对外接口的变更 **REQUIRED** 遵循以下规则：

- **非破坏性变更**（新增可选参数、新增返回字段、新增接口方法）：直接变更，更新文档
- **破坏性变更**（删除/重命名接口、修改参数语义、改变返回值结构）：**REQUIRED** 先将旧接口标记为 deprecated 并提供新接口，在所有调用方迁移完成后移除旧接口
- **破坏性变更的判定标准**：如果调用方不修改代码就无法通过编译或运行时行为发生改变，则为破坏性变更

### 3.2 实体建模

#### 核心实体判定

同时满足以下三个条件的对象为核心实体，**REQUIRED** 在架构文档中描述：

- 出现在 ≥ 2 个关键 Flow 中
- 拥有独立的生命周期（可被创建、修改、删除）
- 承载系统的核心业务语义（去掉它系统无法运行）

**REQUIRED** 描述每个核心实体的：业务概念、关键属性（语义层面）、与其他实体的关系。**PROHIBITED** 写字段类型、数据库列名、序列化格式。

示例：`**Session**：用户的一次对话会话，包含多条消息，归属于创建它的用户。`

#### 实体状态设计

具有 ≥ 3 个业务状态的核心实体，**REQUIRED** 描述其状态生命周期：

- **REQUIRED** 在所属子模块文档中用 Mermaid State Diagram 描述合法的状态转换
- **REQUIRED** 每个状态转换标注触发条件（如"用户支付成功"），不标注实现细节
- **PROHIBITED** 包含系统内部技术状态（如 "db_syncing"、"cache_warming"），只描述业务状态

状态设计回答的问题是："该实体在什么状态下允许什么操作"——这是核心业务规则，不是实现细节。

#### 实体关系图

**REQUIRED** 使用 Mermaid ER 图表示核心实体关系（3 ~ 8 个实体）。关系标注业务语义（如"一个用户拥有多个会话"），**PROHIBITED** 标注数据库外键名或展示字段列表。

### 3.3 设计决策记录（ADR）

每个值得记录的设计决策使用以下 5 列表格格式（前 4 列为 REQUIRED，第 5 列为 OPTIONAL）：

| 决策问题 | 选择 | 放弃的替代方案 | 理由 | 变更条件 |
|---------|------|--------------|------|----------|

字段说明：

- **决策问题**：用疑问句描述面临的选择（如"消息存储用 SQLite 还是文件系统？"）
- **选择**：明确选定的方案
- **放弃的替代方案**：至少列出一个被否定的方案
- **理由**：为什么选这个、不选替代方案
- **变更条件**：什么情况下这个决策应该被重新审视（可不填）

**REQUIRED**：每个 ADR 必须包含前 4 个字段。  
**PROHIBITED**：决策记录中写实现细节、代码示例、参数配置。

**格式选择：**

- 理由可在一句话内表达的决策：使用上述表格格式
- 理由需要多段论述或替代方案需要详细对比的决策：展开为独立小节，但 **REQUIRED** 包含相同的 4 个字段（决策问题、选择、放弃的替代方案、理由），每个字段作为一个段落

## Part 4：文档规范

### 4.1 文档分类

架构文档分为**设计文档**和**参考文档**两类，核心区别在于**更新驱动力不同**：

| 维度 | 设计文档 | 参考文档 |
|------|---------|---------|
| **回答的问题** | "为什么这样设计" | "当前是什么样" |
| **更新驱动** | 设计决策变更时更新 | 代码/配置变更时同步更新 |
| **与代码的关系** | 代码实现设计文档的意图，代码重构不影响设计文档 | 代码变了参考文档必须跟着变，否则删除 |
| **存放位置** | `docs/architecture/` 根目录 | `docs/architecture/reference/` |

设计文档进一步分为两种：

- **总览文档**（Part 4.2）：全局视角，描述系统整体的原则、边界、实体关系。每个系统只有一个
- **子模块文档**（Part 4.3）：单模块视角，描述一个模块的职责、边界、决策。每个模块至多一个

**判断依据**：当需要写入一条新内容时，用以下问题判断归属——

1. 这条内容是否会因代码重构（不改接口）而过时？→ **是**则归参考文档，**否**则归设计文档
2. 若归设计文档：这条内容影响全局还是单个模块？→ **全局**归总览文档，**单模块**归子模块文档

**PROHIBITED** 两类文档混放在同一目录层级。

**篇幅控制**（适用于所有设计文档）：若文档过长，检查是否混入了子模块细节（应移至子模块文档）、实现细节（应移至参考文档或代码）、或过于详细的图表（应用更高层级的抽象替代）。

---

### 4.2 总览文档（Overview Doc）

**定位**：从全局视角描述整个系统或子系统，是架构文档的入口。每个系统只有一个总览文档。

**REQUIRED 章节（按顺序）**：

#### § 系统目标与约束

- 一句话描述系统是什么、为谁服务
- 列出核心约束（**REQUIRED** 每条约束可验证——能回答"是否满足"）
- **PROHIBITED**：路线图、版本计划、"未来考虑"

#### § 核心设计原则

- 3 ~ 5 条，每条包含：名称 + 一句规则描述 + 一句理由
- **REQUIRED** 原则应能指导子模块的具体决策
- **PROHIBITED**：超过 5 条原则、无理由的原则

#### § 关键设计决策

- 使用 Part 3.3 的 ADR 表格格式
- 仅列影响全局的决策，3 ~ 7 条
- **PROHIBITED**：单个模块内部的决策（归属对应子模块文档）

#### § 边界划分

- 用 ASCII 图或层级列表展示主要模块/层
- **REQUIRED** 每个模块一句话说明职责
- 箭头表示数据流方向
- **PROHIBITED**：展示调用栈细节、方法名、参数列表

#### § 核心实体关系

- 按 Part 3.2 的格式描述每个核心实体
- **REQUIRED** 使用 Mermaid ER 图展示实体关系
- **PROHIBITED**：字段列表、类型定义

#### § 整体流程

- 1 ~ 2 个端到端主路径，按 Part 5.3 图表选型规则选择表达方式
- **PROHIBITED**：异常路径、边缘 case、错误处理步骤

#### § 部署架构（OPTIONAL）

描述运行时进程拓扑、部署方式

#### § 安全架构（OPTIONAL）

描述信任边界、权限模型、密钥管理方式

---

### 4.3 子模块文档（Sub-module Doc）

**定位**：围绕单一模块职责展开，解释"这个模块做什么、为什么这样做"。

**REQUIRED 章节（按顺序）**：

#### § 职责定位

格式固定：`[模块名] 负责 [做什么]，不负责 [不做什么]。`

- **REQUIRED** 包含"不负责什么"——这是边界声明，不可省略
- 字数建议控制在 60 字以内，确保简洁清晰
- 这句话应能回答："我该往哪个模块加这个功能？"

#### § 核心原则（OPTIONAL）

- 本模块特有的、补充全局原则的约束，≤ 3 条
- 若本模块无额外约束，省略此节

#### § 边界与实体

**REQUIRED** 包含以下四项：

- **输入**：接收什么（接口名称 + 一句语义说明）
- **输出**：产生什么（接口名称 + 一句语义说明）
- **核心实体**：本模块管理的实体，按 Part 3.2 格式描述
- **错误边界**：本模块捕获/翻译哪类错误，向上层暴露什么错误（一句话说明，不写具体处理步骤）

**PROHIBITED**：字段详细列表、序列化格式、数据库 schema

#### § 关键流程

- 描述本模块的主执行路径，按 Part 5.3 图表选型规则选择表达方式
- **PROHIBITED**：异常处理步骤、错误路径、重试逻辑

#### § 设计决策与权衡

- 使用 Part 3.3 的 ADR 表格格式，范围限于本模块
- **REQUIRED**：每个 ADR 必须有"放弃的替代方案"
- 这是子模块文档中最重要的节——记录"为什么"，而非"是什么"

---

### 4.4 参考文档（Reference Doc）

**定位**：描述客观现状，不记录设计意图。典型例子：代码目录结构、数据库表索引。

**通用规则：**

- **REQUIRED** 只描述当前状态，不写"为什么这样组织"
- **REQUIRED** 代码重组时同步更新，无法保持同步则删除
- **PROHIBITED** 写设计决策、设计原则
- **PROHIBITED** 放在设计文档目录中（必须在 `reference/` 子目录）

**参考文档清单：**

以下参考文档在**对应实体存在时 REQUIRED**，不存在时不创建空文档：

| 文档 | 文件名 | 前置条件 | 触发更新条件 |
|------|--------|---------|-------------|
| 目录结构 | `directory-structure.md` | 始终需要 | 代码目录重组 |
| 依赖清单 | `dependencies.md` | 始终需要 | Cargo.toml / package.json 变更 |
| 配置设计 | `config.md` | 系统存在配置项 | 配置字段增减、默认值变更 |
| 数据库说明 | `database-notes.md` | 系统使用数据库 | 表结构变更、索引变更、归档策略变更 |
| 类型索引 | `type-registry.md` | 存在跨模块接口契约 | 接口契约变更、核心数据结构变更 |

**各类参考文档编写规则：**

**config.md**：

- **REQUIRED** 列出每个配置项的名称、类型、默认值、说明
- **REQUIRED** 说明配置加载顺序（默认值 → 环境变量 → 运行时覆盖）
- **REQUIRED** 包含配置文件格式示例

**database-notes.md**：

- **REQUIRED** 列出所有表名、写入方、索引用途
- **REQUIRED** 说明归档策略（数据生命周期、冷热分离规则）
- **REQUIRED** 说明并发模型（WAL 模式、连接池策略）
- **OPTIONAL** 列出完整 DDL（若 DDL 较短）；否则链接到迁移文件

**type-registry.md**：

- **REQUIRED** 列出所有跨模块接口契约（trait）的位置和关键约束
- **REQUIRED** 列出核心数据结构的位置、用途和关键约束
- **OPTIONAL** 包含关键接口签名（单块 ≤ 10 行）
- **PROHIBITED** 写完整结构体字段定义（直接查阅代码）

### 4.5 目录结构与命名

**REQUIRED** 目录结构：

```
docs/architecture/
├── 00-overview.md              # 总览文档（唯一，不按版本拆分）
├── NN-kebab-case.md            # 主题子模块文档（NN = 两位数字，表示阅读顺序）
├── NN.MM-kebab-case.md         # 子文档（MM = 子序号）
└── reference/
    ├── directory-structure.md  # 参考文档
    └── ...
```

**命名规则：**

- 文件名全小写，单词间用连字符 `-` 分隔
- 数字编号仅表示阅读顺序，不表示版本
- **PROHIBITED** 使用 `v1/`、`v2/`、`old-` 等版本前缀目录——版本通过文档状态标注表达
- **REQUIRED** 同一主题只允许一个 `active` 文档存在

### 4.6 文档间关联与导航

**REQUIRED** `00-overview.md` 作为架构文档的导航入口，列出所有 `active` 文档的链接与一句话说明。`deprecated/` 文件夹中的文档不出现在导航中。

**交叉引用规则：**

- **REQUIRED** 引用其他文档内容时使用相对链接指向原文，**PROHIBITED** 复制大段内容
- **REQUIRED** 提及另一个模块时附带指向该模块文档的链接

## Part 5：写作规则

### 5.1 文档状态标注

**REQUIRED** 每个文档（active/draft）头部包含状态标注：

```markdown
> **Status**: `draft` | `active`
```

废弃文档移入 `deprecated/` 文件夹，文件头部添加：

```markdown
> **Superseded by**: [链接]
```

**状态规则：**

- `draft → active`：自检清单（Part 6）和 `docs-review-spec` 全部通过后标记为 `active`
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹并添加 `Superseded by` 链接；无替代文档则直接删除
- Agent 修改涉及 ≥ 2 个模块的代码时，**REQUIRED** 检查受影响模块的架构文档是否仍然准确
- 参考文档在对应代码变更时同步更新；无法保持同步则删除

---

### 5.2 语言精确性

**PROHIBITED** 在约束/原则/决策中使用模糊词汇：可能、一般来说、通常、大概、某些情况下、视情况而定。（列举示例时使用"等"是允许的。）

**REQUIRED** 写作格式：

- 以系统/模块为主语，使用主动语态
- 约束必须可验证（能回答"是否满足"）
- 原则 = 名称 + 规则 + 理由；决策 = 选了什么 + 没选什么 + 理由

**示例：**

| ✗ | ✓ |
|---|---|
| "该模块通常负责处理用户请求" | "该模块接收用户请求，验证权限，转发至对应 Service" |
| "数据可能存储在多个地方" | "数据存储在 X 模块；Y 模块通过 X 的接口读取，不直接访问存储" |

---

### 5.3 代码与图表规则

**代码规则：**

- **PROHIBITED** 在架构文档中写伪代码、算法实现、控制流代码
- **OPTIONAL** 接口契约可用语言无关的类型签名表示（如 `fn process(input: Request) -> Response`），单块 ≤ 10 行
- **PROHIBITED** 在类型签名中写方法体、错误处理、日志调用

**图表选型：**

- **ASCII 图**用于：边界划分、部署拓扑、端到端流程
- **Mermaid** 用于：实体关系（ER）、状态转换（State）、模块交互时序（Sequence）、分支密集流程（Flowchart）
- **编号步骤**用于：≤ 5 步的简单线性流程

---

### 5.4 内容边界

下表定义哪些内容属于架构文档，哪些属于实现文档：

| 内容类型 | 架构文档 | 实现文档 |
|---------|---------|---------|
| 模块职责定义 | REQUIRED | 可选 |
| 接口契约（类型签名）| 可选 | REQUIRED |
| 核心实体语义描述 | REQUIRED | 可选 |
| 设计决策与理由 | REQUIRED | 可选 |
| 错误边界（哪个模块捕获/翻译哪类错误）| REQUIRED | 可选 |
| 错误处理路径（重试、降级、回滚的具体步骤）| PROHIBITED | REQUIRED |
| 算法步骤（> 3 步）| PROHIBITED | REQUIRED |
| 具体参数值/阈值 | PROHIBITED | REQUIRED |
| 代码实现 | PROHIBITED | REQUIRED |
| 测试用例/场景 | PROHIBITED | REQUIRED |

## Part 6：自检清单

Agent 在生成或审查架构文档后，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。存在未通过项时，**REQUIRED** 修正对应内容后重新检查。

### 6.1 总览文档

- [ ] 包含：一句话系统描述、可验证约束、有理由的设计原则、模块职责说明？
- [ ] 核心实体有 Mermaid ER 图，无字段列表？
- [ ] 流程只包含主路径，图表选型符合 Part 5.3？
- [ ] 边界划分声明了跨切关注点及其实现策略？
- [ ] 模块间依赖构成 DAG？

### 6.2 子模块文档

- [ ] 职责定位包含"不负责什么"？
- [ ] 输入/输出有语义说明，错误边界已声明？
- [ ] ≥ 3 个业务状态的实体有 State Diagram？
- [ ] 每个 ADR 包含"放弃的替代方案"？

### 6.3 通用质量

- [ ] Status 标注存在（`draft` 或 `active`）？
- [ ] 无模糊词汇（可能、通常、大概等）？
- [ ] 无实现细节（算法、代码、配置参数）？
- [ ] 文档间引用使用相对链接，无大段内容复制？
- [ ] `00-overview.md` 导航与文档状态一致（仅列 `active`）？

### 6.4 参考文档（附加）

- [ ] 仅在对应实体存在时创建（无空文档）？
- [ ] 各参考文档内容符合 Part 4.4 编写规则？
- [ ] 无设计决策、设计原则内容？

## Agent 执行工作流

Agent 处理架构文档任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不需要写/更新文档，结束
   └── 满足触发条件 → 继续

2. 判断文档类型（Part 4.1）
   ├── 设计文档
   │   ├── 全局视角 → 总览文档（Part 4.2）
   │   └── 单模块视角 → 子模块文档（Part 4.3）
   └── 客观现状 → 参考文档（Part 4.4）

3. 按对应章节结构写作
   ├── 遵守语言精确性规则（Part 5.2）
   ├── 根据场景选择合适的图表（Part 5.3）
   └── 遵守内容边界规则（Part 5.4）

4. 执行自检清单（Part 6）
   ├── 所有项通过 → 继续
   └── 存在未通过项 → 修正后重新检查

5. 执行交付前质量门（docs-review-spec）
   ├── 读取 references/docs-review-spec.md
   ├── 执行合规检查与润色优化
   ├── 所有项通过 → 输出文档
   └── 存在未通过项 → 修正后重新检查
```
