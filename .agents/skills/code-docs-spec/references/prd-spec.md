# 功能需求规格规范

> 本规范适用于软件功能的需求描述与范围界定。  
> 规范同时作为 Agent 可执行的规则集使用——每条规则均标注 **REQUIRED** / **PROHIBITED** / **OPTIONAL**，Agent 可机械地对照执行与自检。

## Part 1：术语表

本规范中所有术语以此定义为准，Agent 和人均使用同一词汇：

| 术语 | 定义 |
|------|------|
| **Feature（特性）** | 用户可感知的、可独立交付的功能单元 |
| **User Story（用户故事）** | 从用户行为角度描述一个功能点的最小单元，格式"作为 [角色]，我希望 [行为]，以便 [目的]" |
| **Acceptance Criteria（验收标准）** | 定义功能完成条件的可验证检查项，每条能独立判断是否满足 |
| **In Scope** | 本文档明确承诺实现的功能集合 |
| **Out of Scope** | 明确排除、不在本文档范围内的内容；每条必须附有排除理由 |
| **Non-functional Requirement（非功能需求）** | 对系统行为的约束，与具体功能无关（性能、安全、兼容性等），必须有量化阈值 |
| **Constraint（约束）** | 不可违背的限制，违背导致需求无效或系统不可用 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

## Part 2：触发条件

功能需求文档的更新判断基于一条核心原则：**用户可见行为或功能范围发生变更时写文档，实现细节变更时不写**。

**REQUIRED** 新建或更新功能需求文档的条件（满足任一即触发）：

- 新增 Feature 且用户可以感知（有新的用户操作入口或系统响应行为）
- 已有功能的行为发生变更（同一操作触发不同的系统响应）
- 功能范围发生变更（原 Out of Scope 变为 In Scope，或反之）
- 跨 ≥ 3 个模块边界的新交互流程
- 已有文档中记录的验收标准被推翻

**PROHIBITED** 触发功能需求文档更新的变更：

- 纯技术实现方案调整（用户不可见）
- 性能优化（不改变功能行为）
- Bug 修复（恢复已有验收标准，不引入新行为）
- UI 视觉调整（不改变功能逻辑）

## Part 3：需求分析方法

### 3.1 功能粒度判断

判断一个需求应该写为独立的 Feature 文档还是归入已有文档中的 User Story：

**独立 Feature 文档的条件**（满足任一）：

- 涉及多个 User Story 且这些 Story 共享同一用户目标
- 引入新的用户角色或系统角色
- 跨越 ≥ 2 个模块边界的功能流程

**归入已有文档的条件**：

- 在已有 Feature 范围内的扩展（同一用户目标下的新路径）
- 同一交互流程的新变体（如"登录"新增手机号登录方式）

**PROHIBITED 的粒度划分方式**：

- 按技术实现拆分（如"前端需求"/"后端需求"各一个文档）
- 按开发阶段拆分（如"第一期"/"第二期"各一个文档）
- 将单个 User Story 升格为独立 Feature 文档

### 3.2 需求边界界定

确定 In Scope vs Out of Scope 的判断原则：

**属于 In Scope 的内容**：

- 用户完成核心任务所必须经过的每一个交互步骤
- 核心任务失败时系统必须给出的响应（错误提示、降级行为）
- 与该 Feature 直接关联的权限模型变更

**属于 Out of Scope 的内容**：

- 与核心任务无关的辅助功能（即使技术上相关）
- 其他 Feature 文档已覆盖的功能
- 技术实现细节（如何做，而非做什么）

**REQUIRED** Out of Scope 每条必须说明排除原因——"不做"是一个设计决策，原因必须记录。

**PROHIBITED** 用"暂不支持"、"后续考虑"等无决策信息的描述代替明确排除。

### 3.3 验收标准设计

验收标准是功能完成的判断依据。**REQUIRED** 每条验收标准满足以下三个条件：

1. **可机械验证**：能通过确定性测试判断是否满足，不依赖主观判断
2. **无歧义**：不同人独立阅读得出相同理解
3. **独立**：不依赖其他验收标准的执行结果

**REQUIRED** 使用 Given-When-Then 格式或检查项列表：

```
Given [明确的前置状态]
When [用户执行的具体操作]
Then [系统产生的可观测结果]
```

或

```
- [ ] [可验证的系统状态或行为描述]
```

判断一条验收标准是否合格：能否直接将它转化为自动化测试？→ 是则合格，否则需要重写。

**PROHIBITED** 验收标准中的内容：

- 主观描述（"界面美观"、"用户体验良好"）
- 实现细节（"调用 X API"、"写入 Y 数据库"）
- 条件嵌套（"如果…则…否则…"的复合描述——应拆分为多条）

### 3.4 优先级判断

**REQUIRED** 每个 User Story 使用以下优先级定义：

| 优先级 | 定义 | 判断标准 |
|--------|------|---------|
| **P0** | 没有此功能则 Feature 无法使用 | 移除此 Story，用户能否完成核心任务？→ 不能则 P0 |
| **P1** | 核心路径必须包含 | 此 Story 不存在时，是否有替代方案？→ 无替代则 P1 |
| **P2** | 提升体验但存在替代方案 | 用户能否通过其他方式达到相同目的？→ 能则 P2 |
| **P3** | 增值功能，无直接业务必要性 | 去掉后用户是否有明显感知影响？→ 无则 P3 |

### 3.5 非功能需求提取

判断是否需要非功能需求章节（满足任一则 **REQUIRED**）：

- 功能涉及并发用户操作（并发量约束）
- 功能处理敏感数据（安全约束）
- 功能对响应时间有用户可感知的影响（性能约束）
- 功能需在特定平台/设备运行（兼容性约束）

**REQUIRED** 所有非功能需求必须有量化阈值，能回答"是否满足"。

**PROHIBITED** 内容：

- 业务 KPI（转化率、留存率、NPS 等）——这些不是需求约束，是业务结果
- 无量化阈值的约束（"系统应该快速响应"）

## Part 4：文档规范

### 4.1 功能需求文档标准结构

**REQUIRED 章节（按顺序）**：

#### § 背景与目标

- **背景**：为什么要实现这个 Feature（问题或机会描述，1~3 句话）
- **目标**：本文档要解决的核心问题，1~2 句话
- **PROHIBITED**：实现方案、技术细节、"未来考虑"、业务 KPI

#### § 功能描述

按 User Story 组织，每个 Story **REQUIRED** 包含：

- **Story 描述**（"作为 [角色]，我希望 [行为]，以便 [目的]" 格式）
- **验收标准**（Part 3.3 的 Given-When-Then 或检查项格式）
- **优先级**（P0/P1/P2/P3，按 Part 3.4 定义）

**REQUIRED** 同一用户操作流程中的 Stories 按流程顺序排列，不按优先级排列。

#### § 范围界定

**REQUIRED** 包含：

- **In Scope**：本文档承诺实现的功能列表
- **Out of Scope**：明确排除的内容及排除理由（每条必须有理由）

**PROHIBITED** Out of Scope 使用"暂不支持"、"后续考虑"等无决策信息的描述。

#### § 非功能需求（按 Part 3.5 规则判断是否 REQUIRED）

- 性能约束（具体量化阈值）
- 安全约束（数据处理规则、权限模型）
- 兼容性约束（支持的平台/版本范围）

### 4.2 文档篇幅控制

- **单一 User Story 功能**：背景与目标 + 功能描述（单 Story）+ 范围界定，约 1~2 页
- **多 Story 的 Feature**：完整 4 章，约 3~6 页
- **PROHIBITED** 为凑篇幅填充无关内容

### 4.3 目录结构与命名

**REQUIRED** 目录结构：

```
docs/prd/
├── 00-overview.md               # PRD 总览文档（唯一，导航入口）
├── NN-{feature}.md              # Feature 文档（两位数字序号，表示阅读顺序）
├── NN.MM-{sub-feature}.md       # 子 Feature 文档（NN.MM 格式）
├── deprecated/                  # 有替代文档的废弃文档
└── reference/                   # 参考文档
```

**命名规则**：

- 文件名全小写，单词间用连字符 `-` 分隔
- 数字编号仅表示阅读顺序，不表示版本
- **PROHIBITED** 使用 `v1/`、`v2/`、`old-` 等版本标记
- **REQUIRED** 同一主题只允许一个 `active` 文档存在
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹；无替代文档则直接删除

### 4.4 文档间关联

**REQUIRED** `00-overview.md` 作为 PRD 的导航入口，列出所有 `active` 文档的链接与一句话说明。`deprecated/` 文件夹中的文档不出现在导航中。

**交叉引用规则**：

- **REQUIRED** Feature 文档引用关联的架构文档（功能对应的模块文档）
- **REQUIRED** 子 Feature 文档引用父 Feature 文档
- **PROHIBITED** 在 Feature 文档中复制架构文档内容

## Part 5：写作规则

### 5.1 文档状态标注

**REQUIRED** 每个文档头部包含状态标注：

```markdown
> **Status**: `draft` | `active`
```

废弃文档移入 `deprecated/` 文件夹，文件头部添加：

```markdown
> **Superseded by**: [链接]
```

**状态规则**：

- `draft → active`：自检清单（Part 6）和 `docs-review-spec` 全部通过后标记为 `active`
- 废弃文档：有替代文档则移入 `deprecated/` 文件夹并添加 `Superseded by` 链接；无替代文档则直接删除

### 5.2 语言精确性

**PROHIBITED** 在需求描述和验收标准中使用模糊词汇：可能、大概、应该、通常、视情况而定、尽量、适当。

**REQUIRED** 使用确定性语言：

| ✗ | ✓ |
|---|---|
| "页面加载应该很快" | "首屏加载时间 ≤ 2s（4G 网络）" |
| "支持主流浏览器" | "支持 Chrome 90+、Safari 14+、Edge 90+" |
| "用户方便操作" | "核心操作步骤 ≤ 3 步" |
| "数据需要保护" | "数据传输使用 TLS 1.2+，存储使用 AES-256 加密" |

### 5.3 内容边界

下表定义哪些内容属于功能需求文档，哪些属于其他文档：

| 内容类型 | 功能需求文档 | 其他文档 |
|---------|------------|---------|
| 用户可见行为描述 | REQUIRED | 可选 |
| 验收标准 | REQUIRED | PROHIBITED |
| 功能范围（In/Out Scope）| REQUIRED | PROHIBITED |
| 优先级 | REQUIRED | 可选 |
| 非功能需求（有量化阈值）| 按 Part 3.5 | 可选 |
| 技术实现方案 | PROHIBITED | REQUIRED（架构文档） |
| 数据库设计 | PROHIBITED | REQUIRED（架构文档） |
| 接口定义 | PROHIBITED | REQUIRED（架构文档） |
| 测试用例 | PROHIBITED | REQUIRED（测试文档） |
| 业务 KPI（转化率、NPS）| PROHIBITED | 不属于设计文档体系 |

## Part 6：自检清单

Agent 在生成或审查功能需求文档后，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。存在未通过项时，**REQUIRED** 修正对应内容后重新检查。

### 6.1 内容完整性

- [ ] 背景说明了为什么要做这个 Feature（问题或机会），无实现方案？
- [ ] 每个 User Story 使用"作为/我希望/以便"三段式格式？
- [ ] 每个 User Story 有验收标准？
- [ ] 所有需求标注了优先级（P0/P1/P2/P3）？
- [ ] Out of Scope 每条都有排除理由？

### 6.2 质量检查

- [ ] Status 标注存在（`draft` 或 `active`）？
- [ ] 无模糊词汇（可能、大概、应该、通常等）？
- [ ] 每条验收标准可独立机械验证（无主观判断、无实现细节、无条件嵌套）？
- [ ] 无技术实现细节（数据库、API、代码）？
- [ ] 非功能需求（如有）有具体的量化阈值？
- [ ] 文档间引用使用相对链接，无大段内容复制？
- [ ] `00-overview.md` 导航与文档状态一致（仅列 `active`）？

## Agent 执行工作流

Agent 处理功能需求文档任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不需要写/更新文档，结束
   └── 满足触发条件 → 继续

2. 判断功能粒度（Part 3.1）
   ├── 满足独立 Feature 文档条件 → 新建文档
   └── 属于已有 Feature 范围 → 追加 User Story 到已有文档

3. 界定需求范围（Part 3.2）
   ├── 确定 In Scope 列表
   └── 明确 Out of Scope 并记录每条排除理由

4. 按文档结构写作（Part 4.1）
   ├── 使用 Part 3.3 设计验收标准（检查：能否转化为自动化测试？）
   ├── 使用 Part 3.4 标注优先级（逐 Story 对照判断标准）
   ├── 按 Part 3.5 判断是否需要非功能需求章节
   └── 遵守语言精确性规则（Part 5.2）

5. 执行自检清单（Part 6）
   ├── 所有项通过 → 继续
   └── 存在未通过项 → 修正后重新检查

6. 执行交付前质量门（docs-review-spec）
   ├── 读取 references/docs-review-spec.md
   ├── 执行合规检查与润色优化
   ├── 所有项通过 → 输出文档
   └── 存在未通过项 → 修正后重新检查
```
